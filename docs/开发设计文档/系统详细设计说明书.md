## 1. 引言

### 1.1 编写目的
本文档在《系统概要设计说明书》基础上，对 UrbanMicro-CAD 系统进行**面向编码实现的详细设计**。采用 DDD（领域驱动设计）方法论划分限界上下文，使用 Mermaid 绘制类图、包图与时序图，定义完整的数据库 Schema，为开发团队提供可直接落地的编码蓝图。

### 1.2 术语与缩写
| 缩写 | 全称 | 含义 |
|:---|:---|:---|
| DDD | Domain-Driven Design | 领域驱动设计 |
| BC | Bounded Context | 限界上下文 |
| AR | Aggregate Root | 聚合根 |
| VO | Value Object | 值对象 |
| DM | Domain Model | 领域模型 |
| HE | Half-Edge | 半边数据结构 |
| IDM | Intelligent Driver Model | 智能驾驶员跟驰模型 |
| MOBIL | Minimizing Overall Braking Induced by Lane Changes | 最小化变道制动模型 |
| OD | Origin-Destination | 起讫点 |
| ECS | Entity-Component-System | 实体组件系统 |
| LOS | Level of Service | 服务水平评级 |
| SSOT | Single Source of Truth | 单一数据源 |

### 1.3 设计约束
- 前端：Vue 3.4+ / TypeScript 5+ / Vite 5+ / Pinia 2+ / Three.js 0.160+
- 后端：Spring Boot 3.2+ / Java 21 / MyBatis-Plus 3.5+ / PostgreSQL 14+ (PostGIS)
- 严禁使用 `any` / `@ts-ignore`；所有 Geometry/Material/Texture 必须 `.dispose()`
- 大量对象必须使用 `InstancedMesh`；重计算必须使用 Web Worker + Comlink

## 4. 核心模块详细类图

### 4.1 路网建模子系统 (FR1)

#### 4.1.1 功能详细说明

**对标机制**：原版道路建设工具、`Network Multitool`、`Fine Road Anarchy 2`

本子系统是整个系统的数据基石，负责路网拓扑的创建、编辑与维护。所有后续的交规配置、仿真运行与评估分析均依赖本子系统产出的路网数据。

| 功能编号 | 功能名称 | 详细描述 | 关键实现机制 |
|:---|:---|:---|:---|
| FR1.1 | 道路绘制交互与状态机 | 绘制状态机（IDLE→DRAWING→PREVIEW→CONFIRMED）；连续绘制模式（终点自动成为下一段起点）；三种绘制模式（直线/曲线/自由）；实时3D预览（半透明路面网格）；测量标注（长度/坡度/角度）；智能吸附（10px半径）与角度约束（15°倍数） | `DrawingState` 状态机驱动；预览使用 `MeshBasicMaterial(opacity:0.5)` + 简化几何管线；测量标注使用 `CSS2DRenderer`；吸附检测基于2D空间哈希；角度约束通过向量点积 |
| FR1.2 | 高程控制与坡度约束 | 通过 PageUp/PageDown 调整端点Z值；三种高程模式切换（F1地面/F2高架/F3隧道）各有独立最大坡度限制；Anarchy模式解除所有坡度限制；自动坡道过渡；坡度可视化（绿→黄→红着色） | `ElevationProfile` 记录起终点Z值+高程模式+`RampTransition`；坡度超限时预览路面变红阻止确认；坡度着色通过 `ShaderMaterial` Uniform 驱动 |
| FR1.3 | 道路升级工具 | "升级刷"原位无损升级横断面；升级时保留TM:PE规则；**降级保护**：降级时校验车道连接器冲突，有冲突则阻止 | `RoadSegment.upgrade()` 替换 profile 后触发 Worker 重算；规则通过 laneId 映射保留；降级前调用 `TrafficRuleStore.validateDowngrade()` |
| FR1.4 | 横断面模板库 | 内置基础路、干道、高速公路、土路等模板。支持复合断面：带自行车道、公交专用道、电车轨道、中央绿化带/隔音屏障 | `CrossSectionProfile` 值对象包含 `LaneConfig[]` + `MedianConfig` + `SidewalkConfig`，模板即预定义的 Profile 实例 |
| FR1.5 | 动态拓扑自愈 | 新路段交叉自动打断生成节点；删除路段时自动重算图论拓扑（十字路口→丁字路口自动退化）；半边拓扑实时维护 | `TopologyHealingService` 监听路段增删事件，执行交叉检测→节点打断→半边重连→边界重算 |
| FR1.6 | 连续绘制与对齐工具 | 延续绘制：选中路段端点后自动以该端点横断面为模板延续绘制；平行路绘制：按P键进入平行路模式，输入偏移距离自动生成平行路段；对齐参考线：绘制时自动显示平行/垂直对齐辅助线 | 延续绘制复用 `CrossSectionProfile`；平行路通过 `clipper-lib` 对中心线做固定距离偏移；对齐辅助线通过向量点积计算最近15°倍数角度 |
| FR1.7 | 推土机与删除工具 | 删除路段时弹出确认提示（含级联影响说明）；支持框选批量删除；通过 `DeleteRoadCommand` 支持撤销/重做 | `DeleteRoadCommand` 保存被删路段完整快照用于 undo；级联影响通过 `TrafficRuleStore` 查询关联规则数量 |
| FR1.8 | 自定义断面编辑器 | 逐条添加/删除车道，独立设置方向(FORWARD/BACKWARD/BOTH)/类型(CAR/BUS/BIKE/TRAM)/宽度(2.5~4.5m)；实时2D横截面预览；支持非对称断面(2+4等)；保存为用户模板；SetCrossSectionCommand支持撤销 | `CrossSectionEditorStore` 管理编辑状态；`clipper-lib` 根据direction决定偏移方向(FORWARD偏左/BACKWARD偏右)；用户模板存入`prj_template` |
| FR1.9 | 测量标注与夹角标注 | 绘制路段时实时标注长度(m)、坡度(%)、高程模式名；选中交叉口节点时标注相连路段间的夹角(°)；夹角标注显示在节点旁悬浮 Sprite，列出所有相邻路段对的角度；MEASURE 工具支持点击两个节点测量距离 | 长度/坡度标注使用 `THREE.Sprite`(Canvas 绘制圆角矩形背景+文字)；夹角计算通过向量点积 `acos((v1·v2)/(|v1|·|v2|))` 并转换为度数；夹角标注数据结构 `AngleAnnotation { nodeId, pairs: AnglePair[] }`，`AnglePair { fromSegmentId, toSegmentId, angleDeg }`；标注跟随节点移动自动更新；MEASURE 工具通过 `MeasurementCommand` 支持撤销 |

**核心数据流**：用户绘制 → `EditorStateStore.drawRoad()` → `DrawRoadCommand.execute()` → `RoadNetworkStore.addSegment()` → `GeometryWorker`（offset→union→triangulate）→ `TopologyHealingService.healOnSegmentAdd()` → Pinia 响应式更新 → Three.js 视口重绘

### 4.2 枢纽节点微调子系统 (FR2)

#### 4.2.1 功能详细说明

**对标机制**：`Move It`、`Node Controller`

本子系统允许用户对路网中已生成的交叉口节点进行精细化三维调整，解决自动生成的交叉口边界不够贴合实际渠化需求的问题。微调操作通过 Gizmo 控件实现所见即所得的交互体验。

| 功能编号 | 功能名称 | 详细描述 | 关键实现机制 |
|:---|:---|:---|:---|
| FR2.1 | 多边形边界自定义 | 允许用户直接拖拽交叉口多边形边界的顶点，将其编辑为任意形状以贴合畸形渠化路口（如斜交路口、喇叭口） | `PolygonBoundary.moveVertex()` 修改顶点坐标 → `GeometryWorker.triangulate()` 重算三角网格 → Three.js 更新 |
| FR2.2 | 三维曲面融合与切线微调 | 提供 Move It 级 3D 切线手柄（Tangent Handles），拖拽手柄时实时插值计算节点内部顶点高程和法线，消除网格扭曲。支持 Coons Patch 双三次曲面融合 | `TangentHandle` 控制 BezierCurve 方向与长度 → `CoonsPatchBuilder.blendSurface()` 生成光滑曲面 → `BufferGeometry` 更新 |
| FR2.3 | 3D 场景对象微调 | 提供万能 Gizmo 控件（Three.js `TransformControls`），支持对节点、桥墩、标志牌进行三维平移(TRANSLATE)、旋转(ROTATE)、缩放(SCALE) | `GizmoController` 封装 `TransformControls`，通过 `NodeAdjustmentStore` 管理激活状态与模式切换 |

**核心数据流**：用户激活微调工具 → `NodeAdjustmentStore.activateNode()` → Three.js 显示 Gizmo → 用户拖拽手柄 → `TangentHandle.setDirection()` → `CoonsPatchBuilder.blendSurface()` [Comlink RPC] → 返回 `BufferGeometry` → Three.js dispose 旧 Geometry → 应用新 Geometry

**性能保障**：拖拽过程中仅更新控制点预览（不触发重算），`pointerup` 事件触发 Lazy Evaluation 调用 Worker 重算完整网格，保证拖拽中 60FPS。

### 4.3 交规引擎与 TM:PE 规则子系统 (FR3)

#### 4.3.1 功能详细说明

**对标机制**：原版基础交规、`TM:PE`、`Intersection Marking Tool (IMT)`

本子系统是系统"硬核"属性的核心来源，深度对标 TM:PE 模组，提供节点级、车道级的微观交通规则配置能力。所有规则数据序列化为 JSON 绑定在节点/车道实体上，作为仿真引擎的输入驱动严格仿真行为。

| 功能编号 | 功能名称 | 详细描述 | 关键实现机制 |
|:---|:---|:---|:---|
| FR3.1 | 原生基础交规配置 | 提供路口"禁止左转/右转/直行"快捷开关（`TurnRestriction`）；支持切换路口控制模式：无控制(YIELD)/红绿灯(TRAFFIC_LIGHT)/环岛(ROUNDABOUT)；支持开启/关闭人行横道（`crosswalkEnabled`） | `TrafficRuleSet` 聚合根管理 `TurnRestriction[]` + `NodeControlMode` + `crosswalkEnabled`；模式切换触发 `DecalRenderer` 更新斑马线贴花 |
| FR3.2 | 车道连接器工具 | 激活后显示进出口车道锚点，用户拖拽连线生成贝塞尔引导线（`CatmullRomCurve3`）。未连接的进口车道视为死胡同，仿真车辆不会从该车道驶出 | `LaneConnector` 记录 fromLaneId→toLaneId 映射 + `fromAnchor/toAnchor` 锚点 + `guideCurve` 引导曲线；3D 渲染为半透明曲线 |
| FR3.3 | 定时与多步阶信控工具 | 提供多步阶编辑器 (Step Editor)，每步阶设置最小/最大绿灯时间、黄灯过渡时间、全红清空时间。支持绑定"虚拟传感器"实现动态感应延长绿灯 | `TrafficLightController` 管理 `SignalStep[]` + `VirtualSensor[]`；通过 `ISignalStrategy` 策略接口支持固定配时(`FixedTimingStrategy`)和感应控制(`SensorActuatedStrategy`)两种模式 |
| FR3.4 | 车道级限制与标线工具 | 单条车道独立限速、特定车型禁行（`LaneRestriction`）；绘制"实线/虚线"同步修改底层变道权限矩阵与 3D 贴花；绘制"导向箭头"锁定强制方向 | `LaneRestriction` 包含 speedLimit/allowedVehicleTypes/allowLeftChange/allowRightChange/markingType；`DecalRenderer` 使用 `DecalGeometry` 将标线 PNG 投影到路面 |
| FR3.5 | 车道级转向箭头 | 每条车道独立设置允许转向(LEFT/STRAIGHT/RIGHT/U_TURN)；新建路口时根据LaneConnector自动推断箭头；用户手动覆盖；3D箭头贴花渲染；仿真车辆提前选择正确车道 | `LaneArrow` 值对象存储于 `TrafficRuleSet.laneArrows[]`；自动推断逻辑遍历 `LaneConnector` 的 fromLaneId→toSegmentId 方向；MOBIL模型在路口前检查 `LaneArrow.allowedDirections` |

**核心数据流**：用户选择节点 → `TrafficRuleStore.loadRuleSet()` → 用户配置规则 → `EditorStateStore.executeCommand()` → `TrafficRuleStore` 更新规则 → `DecalRenderer` 投影贴花 → `StoreEventBus.emit('traffic-rule:rule-changed')` → `SimulationStore` 订阅更新仿真规则

**规则冲突校验**：`RuleValidationService` 在每次规则变更后自动执行冲突检测（如：禁止左转但车道连接器指向左转出口），冲突时阻止提交并高亮提示。

### 4.4 微观交通仿真子系统 (FR4)

#### 4.4.1 功能详细说明

**对标机制**：原版物理引擎、`TM:PE` (严格模式)

本子系统是系统的"引擎"核心，负责在 Web Worker 中运行微观交通仿真循环。采用 1D 运动学插值策略替代 3D 物理碰撞，将跟驰检测复杂度从 O(N²) 降至 O(N)。所有仿真计算在独立 Worker 线程执行，主线程仅负责 InstancedMesh 矩阵更新。

| 功能编号 | 功能名称 | 详细描述 | 关键实现机制 |
|:---|:---|:---|:---|
| FR4.1 | 严格实线约束与车道保持 | 实线区强制锁定变道权限（`allowLeftChange=false`）；错过路口严禁压实线强行变道，必须将错就错至下一路口掉头 | `SimVehicle` 状态机检查 `LaneRestriction.allowLeftChange/allowRightChange`；实线区 `CHANGING_LANE` 状态不可达 |
| FR4.2 | 车道连接器强制引导 | 车辆进入路口后激活 PID 轨迹跟踪器，严格吸附于 `LaneConnector.guideCurve` 引导线行驶，不会偏移到错误出口 | `PIDTracker.compute()` 计算横向修正量 → 车辆位置沿 `CatmullRomCurve3` 插值 → 实现平滑过弯 |
| FR4.3 | 信控状态机与动态感应 | 红灯平滑减速（IDM 模型自然减速），绿灯按当前车道独立限速起步；虚拟传感器检测到车辆时，`SensorActuatedStrategy` 动态延长绿灯 | `TrafficLightController` 通过 `ISignalStrategy` 策略驱动步阶切换；`VirtualSensor.checkVehicle()` 检测到达车辆 → 触发绿灯延长 |
| FR4.4 | 车型与限速强校验 | 卡车误入禁行路段强制刹停（状态→STOPPED）并调用 `reroute()` 重新寻路；最高速度严格受限于当前车道 `LaneRestriction.speedLimit` | `SimVehicle` 每帧校验 `VehicleType ∈ LaneRestriction.allowedVehicleTypes`；违规则刹停 + `IPathFinder.findPath()` 重新规划 |
| FR4.5 | MOBIL 变道模型 | 虚线区通过 MOBIL 模型评估变道：安全性准则（目标车道后车减速度 ≤ b_safe=3m/s²）；激励性准则（加速度增益 - 礼貌因子×后车减速代价 ≥ p=0.2）；平滑横向过渡（lateralOffset 2秒渐变）；实线区变道判断短路 | `MOBILLaneChangeModel.calculateLaneChangeDecision()` 返回变道决策；`SimVehicle.lateralOffset` 渐变驱动 `InstancedMesh` 矩阵偏移；实线区 `CHANGING_LANE` 状态不可达 |
| FR4.6 | 车辆需求与OD模型 | OD矩阵配置（起讫点→小时交通量）；车型比例配置；泊松分布发车；A*最短路径分配；默认"随机流量"模式 | `ODMatrix` + `VehicleMixConfig` 作为 `SimulationWorld` 初始化参数；泊松分布 `Math.random() < 1 - Math.exp(-lambda * dt)` 判断是否发车 |

**仿真循环（每帧）**：
1. 遍历所有 `SimVehicle`，通过 `SpatialHashGrid.queryNearby()` 获取前车
2. 调用 `ICarFollowModel.calculateAcceleration()` 计算目标加速度（默认 IDM 实现）
3. 调用 `MOBILLaneChangeModel.calculateLaneChangeDecision()` 评估变道决策
4. 更新 speed/progress/lateralOffset，状态机判断（变道/红灯停车/路口寻迹）
5. 车辆到达路段末端时，通过 `IPathFinder` 获取下一车道
6. `VehicleSpawner.tick()` 按泊松分布生成新车辆
7. 更新所有 `SignalState`
8. 打包 `SimulationFrameData`（Float32Array）返回主线程

**性能目标**：500+ 车辆单帧计算 < 10ms（Worker 线程），主线程 InstancedMesh 更新 < 2ms。

**双通道数据架构**：仿真位置数据采用**快通道**——Worker 每帧将 `SimulationFrameData`（Float32Array）通过 `SharedArrayBuffer` 直接传递至主线程，主线程遍历数组更新 `InstancedMesh.setMatrixAt()`，**完全绕过 Pinia 响应式追踪**，避免 500+ 车辆的 Proxy 依赖收集开销；仿真元状态（running/speed/vehicleCount/odMatrix/mixConfig）走**慢通道**——通过 `comlink` RPC 写入 `SimulationStore`，由 Vue UI 面板订阅消费。

### 4.7 命令模式 (Undo/Redo 架构)

#### 4.7.1 功能详细说明

**设计模式**：命令模式 (Command Pattern)

本模块为系统提供全局的撤销/重做能力，确保用户任何误操作均可恢复。同时作为系统级容错机制，当 Worker 线程抛出几何计算异常时，自动调用栈顶命令的 `undo()` 实现状态无损回滚。

| 能力 | 详细描述 | 实现机制 |
|:---|:---|:---|
| 50 步撤销/重做 | 所有用户操作（画线、升级道路、调整高程、配置信控、设置车道限制）均支持至少 50 步的撤销与重做 | `HistoryStack` 维护 `undoStack: ICommand[]` 和 `redoStack: ICommand[]`，`maxSize=50`；新命令 push 时清空 redoStack |
| 命令接口统一 | 所有可撤销操作必须实现 `ICommand` 接口 | `ICommand` 定义 `execute()` / `undo()` / `getDescription()` 三个方法；每个命令内部保存执行前的状态快照用于 undo |
| Worker 异常自动回滚 | 当 GeometryWorker 抛出异常（如多边形自交、Offset 失败）时，系统自动调用栈顶命令的 `undo()` | `DrawRoadCommand.execute()` 内部 try-catch Worker 调用；异常时调用 `HistoryStack.undo()` 自动回滚 |
| 命令调度中心 | `EditorStateStore` 作为唯一命令调度入口，所有操作通过 `executeCommand()` 统一执行 | `EditorStateStore.drawRoad()` / `upgradeRoad()` / `deleteRoad()` / `moveNode()` 均内部创建 Command 并调用 `executeCommand()` |

**已实现的命令类**：

| 命令类 | 触发场景 | execute 行为 | undo 行为 |
|:---|:---|:---|:---|
| `DrawRoadCommand` | 用户绘制新路段 | 创建 `RoadSegment` → Worker offset/union → 拓扑自愈 | 删除 `RoadSegment` → Worker 重算交叉口 → 拓扑回退 |
| `UpgradeRoadCommand` | 用户使用升级刷 | 保存旧 `CrossSectionProfile` → 替换新 Profile → Worker 重算 | 恢复旧 `CrossSectionProfile` → Worker 重算 |
| `SetLaneRestrictionCommand` | 用户设置车道限速/禁行 | 保存旧 `LaneRestriction` → 设置新限制 → Decal 更新 | 恢复旧 `LaneRestriction` → Decal 更新 |
| `MoveNodeCommand` | 用户移动节点位置 | 保存旧 `Point2D` → 更新位置 → Worker 重算边界 | 恢复旧 `Point2D` → Worker 重算边界 |
| `DeleteRoadCommand` | 用户使用推土机删除路段 | 保存被删路段完整快照 + deletedConnectors + deletedRestrictions + deletedTurnRestrictions → 删除 RoadSegment → Worker 重算交叉口 → 删除关联规则 | 恢复被删路段快照 + 恢复 deletedConnectors/Restrictions/TurnRestrictions → Worker 重算交叉口 → 恢复关联规则 |
| `SetLaneConnectorCommand` | 用户添加车道连接器 | 保存旧 LaneConnector[] → 添加新连接器 → 重建引导曲线 → 3D渲染 | 恢复旧 LaneConnector[] → 重建引导曲线 → 3D渲染 |
| `AddSignalStepCommand` | 用户添加信控步阶 | 保存旧 SignalStep[] → 添加新步阶 → validate() | 恢复旧 SignalStep[] → validate() |
| `SetElevationCommand` | 用户修改路段高程 | 保存旧 ElevationProfile → 更新高程 → Worker 重算路面网格 | 恢复旧 ElevationProfile → Worker 重算路面网格 |
| `MoveVertexCommand` | 用户拖拽多边形边界顶点 | 保存旧顶点位置 → 更新顶点坐标 → Worker 重算交叉口多边形 → 3D渲染 | 恢复旧顶点位置 → Worker 重算交叉口多边形 → 3D渲染 |
| `SetTangentDirectionCommand` | 用户拖拽3D切线手柄 | 保存旧切线方向 → 更新切线 → Worker 重算 Coons Patch 曲面 → 3D渲染 | 恢复旧切线方向 → Worker 重算曲面 → 3D渲染 |
| `SetTurnRestrictionCommand` | 用户设置转向限制 | 保存旧 TurnRestriction[] → 添加/移除限制 → RuleValidationService.validate() | 恢复旧 TurnRestriction[] → validate() |
| `SetNodeControlModeCommand` | 用户切换路口控制模式 | 保存旧 controlMode → 更新模式 → 清理/初始化信控/让行规则 | 恢复旧 controlMode → 重建规则 |
| `SetCrossSectionCommand` | 用户在断面编辑器中修改断面并应用 | 保存旧 CrossSectionProfile → 替换为新 profile → Worker 重算路面网格 → 3D渲染 | 恢复旧 CrossSectionProfile → Worker 重算路面网格 → 3D渲染 |
| `SetLaneArrowCommand` | 用户设置车道转向箭头 | 保存旧 LaneArrow → 更新 allowedDirections → RuleValidationService.validate() → DecalRenderer 更新箭头贴花 | 恢复旧 LaneArrow → validate() → DecalRenderer 更新箭头贴花 |

### 5.3 JSONB 数据结构定义

#### 5.3.1 topology_data 结构

```typescript
interface TopologyData {
    nodes: {
        id: string;
        position: { x: number; y: number };
        elevation: number;
        boundary: { x: number; y: number }[];
        controlMode: 'YIELD' | 'TRAFFIC_LIGHT' | 'ROUNDABOUT' | 'NONE';
        connectedSegmentIds: string[];
    }[];
    segments: {
        id: string;
        startNodeId: string;
        endNodeId: string;
        centerLine: { x: number; y: number }[];
        profile: CrossSectionProfile;
        elevation: ElevationProfile;
    }[];
    halfEdges: {  // ⚠ 冗余派生数据：可由 nodes + segments 重建，持久化仅为避免运行时重算
        id: string;
        originNodeId: string;
        twinId: string;
        nextId: string;
        segmentId: string;
        laneIndex: number;
    }[];
}
```

> **注**：`centerLine` 存储贝塞尔曲线采样控制点（Point2D[]），运行时通过 `bezier-js` 重建为 `BezierCurve` 对象。`halfEdges` 为冗余派生数据，可从 `nodes` + `segments` 完整重建，持久化目的仅为避免每次加载时的运行时重算开销。

#### 5.3.2 rule_data 结构

```typescript
interface RuleData {
    ruleSets: {
        nodeId: string;
        nodeControlMode: 'YIELD' | 'TRAFFIC_LIGHT' | 'ROUNDABOUT' | 'NONE';
        crosswalkEnabled: boolean;
        turnRestrictions: TurnRestriction[];
        laneArrows: {
            laneId: string;
            nodeId: string;
            allowedDirections: ('LEFT' | 'STRAIGHT' | 'RIGHT' | 'U_TURN')[];
        }[];
        laneConnectors: {
            id: string;
            fromLaneId: string;
            toLaneId: string;
            fromAnchor: { x: number; y: number };
            toAnchor: { x: number; y: number };
        }[];
        trafficLight: {
            id: string;
            steps: SignalStep[];
            sensors: VirtualSensor[];
            strategy: 'FIXED_TIMING' | 'SENSOR_ACTUATED';
        } | null;
        laneRestrictions: {
            laneId: string;
            restriction: LaneRestriction;
        }[];
    }[];
    odConfig: {
        pairs: {
            fromNodeId: string;
            toNodeId: string;
            volumePerHour: number;
        }[];
        vehicleMix: {
            ratios: {
                type: VehicleType;
                ratio: number;
            }[];
        };
    };
}
```

> **注**：`LaneConnector.guideCurve`（CatmullRomCurve3 引导曲线）为运行时根据 `fromAnchor`/`toAnchor` 自动重建的计算字段，不持久化存储。

`prj_snapshot.snapshot_data` 与 `prj_template.snapshot_data` 共享相同的 JSONB 结构，定义如下：

interface SnapshotData {

## 9. 关键算法伪代码

### 9.1 2.5D 路面网格生成流水线

```
输入: centerLine (贝塞尔曲线), profile (横断面), elevation (高程)
输出: MeshData (Three.js BufferGeometry 数据)

1. 利用 bezier-js 采样中心线为 N 个离散点
2. 对每个采样点计算切线方向
3. 利用 clipper-lib 对中心线做左右 Offset，生成 2D 路面多边形
   // Offset距离 = 车道宽度之和/2 + 人行道宽度，即道路半宽
4. 若有交叉口，对多个路面多边形执行 Union 布尔运算
   // Union消除重叠区域，生成完整交叉口2D边界
5. 利用 d3-delaunay 对 2D 多边形做 Delaunay 三角剖分
   // 三角剖分将多边形转为GPU可渲染的三角形图元
6. 根据端点高程线性插值，为每个顶点赋予 Z 值
   // 2.5D策略：XY平面由几何计算保证正确性，Z轴仅做视觉投影
   // 若存在RampTransition，则按transitionCurve类型插值坡度过渡
7. 计算法线、UV 坐标
   // 法线用于光照计算，UV用于路面纹理/Decal贴花投影
8. 输出 MeshData
```

### 9.2 IDM 跟驰模型

```
输入: vehicle (当前车), leader (前车 | null), speedLimit
输出: acceleration (加速度)

1. v0 = min(vehicle.maxSpeed, speedLimit)     // 期望速度：取车辆最大速度与车道限速的较小值
2. s0 = vehicle.minGap                         // 最小安全间距：静止时与前车保持的最小距离（米）
3. T  = 1.5                                    // 安全车头时距：以当前速度行驶1.5秒的距离作为安全缓冲
4. a  = vehicle.maxAcceleration                // 最大加速度：车辆在自由流状态下的最大加速能力（m/s²）
5. b  = vehicle.comfortableDeceleration        // 舒适减速度：驾驶员习惯的制动减速度（m/s²），通常3-4

6. IF leader == null:                          // 无前车：自由行驶
7.   s_star = s0 + v * T                       // 期望间距 = 最小间距 + 速度×时距
8. ELSE:                                       // 有前车：跟驰状态
9.   delta_v = vehicle.speed - leader.speed    // 相对速度：正值表示正在接近前车
10.  s = leader.progress - vehicle.progress - leader.bodyLength  // 实际车间距（弧长差 - 车身长度）
11.  s_star = s0 + max(0, v*T + v*delta_v/(2*sqrt(a*b)))  // 期望间距：含动态制动项

12. acceleration = a * (1 - (v/v0)^4 - (s_star/s)^2)  // IDM核心公式：自由加速项 - 速度项 - 间距项
13. RETURN clamp(acceleration, -b_max, a)      // 限制加速度范围，防止物理不合理值
```

### 9.3 MOBIL 变道模型

```
输入: vehicle (当前车), leftLane (左车道信息), rightLane (右车道信息)
输出: LaneChangeDecision (是否变道 + 目标车道ID)

1. IF 实线区 (allowLeftChange=false AND allowRightChange=false):
2.   RETURN { shouldChange: false, targetLaneId: null, direction: NONE }

3. p  = 0.2                                    // 礼貌因子：越大越"礼貌"，越不容易变道
4. b_safe = 3.0                                 // 安全减速度阈值：后车最大可接受减速度(m/s²)

5. FOR EACH targetLane IN [leftLane, rightLane]:
6.   IF targetLane 被禁行 (LaneRestriction 禁止变道):
7.     CONTINUE
8.   a_c = IDM.calculateAcceleration(vehicle, targetLane.leader)  // 变道后自身加速度
8.5 IF targetLane.follower 为空:
8.6   decel = 0, a_n_new = 0                    // 无后车，安全性准则自动满足
8.7   GOTO 激励性准则判断
9.   a_n = targetLane.follower 的当前加速度                       // 目标车道后车当前加速度
10.  a_n_new = IDM.calculateAcceleration(targetLane.follower, vehicle)  // 变道后后车新加速度
11.  decel = a_n - a_n_new                       // 后车减速代价
12.  gain = a_c - vehicle.currentAcceleration    // 自身加速度增益
13.  IF a_n_new < -b_safe:                       // 安全性准则不满足（标准MOBIL：后车新加速度不低于-b_safe）
14.    CONTINUE                                  // 变道太危险，跳过此方向
15.  IF gain - p * decel > 0:                    // 激励性准则满足
16.    RETURN { shouldChange: true, targetLaneId: targetLane.id, direction: LEFT/RIGHT }
    // 注：车辆在路口前会检查 LaneArrow.allowedDirections，
    // 若当前车道不允许目标转向，MOBIL 的激励性准则中
    // "自身加速度增益"会额外加上"转向合规奖励"，
    // 使车辆更倾向于变到允许目标转向的车道。

17. RETURN { shouldChange: false, targetLaneId: null, direction: NONE }
```

### 9.4 HCM LOS 分级标准

```
输入: averageDelay (平均控制延误，秒/车)
输出: LOSGrade (A-F)

// HCM (Highway Capacity Manual) 服务水平分级标准
// 延误越低，服务水平越高；F级表示严重拥堵
IF averageDelay <= 10  RETURN A    // 自由流，几乎无延误
IF averageDelay <= 20  RETURN B    // 稳定流，短暂延误
IF averageDelay <= 35  RETURN C    // 稳定流，可接受延误
IF averageDelay <= 55  RETURN D    // 接近不稳定流，可容忍延误
IF averageDelay <= 80  RETURN E    // 不稳定流，显著延误
RETURN F                            // 强制流或崩溃流，严重拥堵
```

| 热力图 | GPU ShaderMaterial + Uniform | 零 CPU 开销 |
| 飞线渲染 | Line2 + UV 动画 | GPU 驱动流动 |
| 几何计算 | Web Worker + Comlink | 主线程不阻塞 |
| 仿真计算 | SpatialHashGrid + 1D 跟驰 | 500车 < 10ms/帧 |
| 拖拽预览 | Lazy Evaluation (pointerup 触发) | 拖拽中 60FPS |
